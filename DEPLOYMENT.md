## 1. Local Development

### Step 1 — Backend

```bash
# From the project root (so src/ is importable):
cd your-project

# Install backend deps (uses the root venv)
pip install -r backend/requirements.txt

# Create .env for the backend (optional — defaults work for local dev)
cp backend/.env.example backend/.env

# Start FastAPI on port 8000
uvicorn backend.main:app --reload --port 8000
```

Verify at: http://localhost:8000/docs (Swagger UI auto-generated by FastAPI)

### Step 2 — Frontend

```bash
cd your-project/frontend

# Install Node deps (Node 18+ required)
npm install

# Create local env file pointing at the local API
echo 'NEXT_PUBLIC_API_URL=http://localhost:8000' > .env.local

# Start Next.js dev server on port 3000
npm run dev
```

Open: http://localhost:3000

> **CORS note:** In development, Next.js proxies `/api/*` to the FastAPI
> backend via `next.config.ts` rewrites, so the browser never sends a
> cross-origin request. The FastAPI CORS middleware is the safety net for
> production.

### Running both simultaneously (single terminal)

```bash
# Install concurrently globally once
npm install -g concurrently

# From project root
concurrently \
  "uvicorn backend.main:app --reload --port 8000" \
  "cd frontend && npm run dev"
```

## 2. Environment Variables

### Backend (`backend/.env`)

```env
# Path overrides (defaults work if you run from project root)
RAW_DATA_PATH=data/GlobalWeatherRepository.csv
PROCESSED_DATA_PATH=data/processed_weather_data.csv

# Comma-separated list of allowed CORS origins
# Add your Vercel URL here once deployed
CORS_ORIGINS=http://localhost:3000,https://your-app.vercel.app
```

### Frontend (`frontend/.env.local`)

```env
# Local development — points at local FastAPI
NEXT_PUBLIC_API_URL=http://localhost:8000
```

```env
# Production — set this in Vercel dashboard as an environment variable
NEXT_PUBLIC_API_URL=https://your-fastapi-backend.your-host.com
```

## 3. Deploying the Frontend to Vercel

### Step 1 — Push to GitHub

```bash
git init          # if not already a git repo
git add .
git commit -m "feat: full-stack weather forecasting dashboard"
git remote add origin https://github.com/<you>/weather-forecast.git
git push -u origin main
```

### Step 2 — Import into Vercel

1. Go to https://vercel.com/new
2. Import your GitHub repository
3. Set the **Root Directory** to `frontend`
4. Vercel auto-detects Next.js — no build command changes needed

### Step 3 — Set environment variables in Vercel

In the Vercel project dashboard → Settings → Environment Variables:

| Name | Value |
|---|---|
| `NEXT_PUBLIC_API_URL` | `https://your-fastapi-host.com` |

### Step 4 — Deploy

Click **Deploy**. Vercel builds and deploys the Next.js frontend. Every `git push` to `main` triggers an automatic redeployment.

## 4. Deploying the Backend

Because the ML models are too large for Vercel's serverless functions (500 MB limit), the FastAPI backend must be deployed to a VM-based host. Recommended options:

### Option A — Railway (easiest, no DevOps)

```bash
# Install Railway CLI
npm install -g @railway/cli
railway login

# From the project root
railway init
railway up --service backend
```

Set environment variables in the Railway dashboard.

### Option B — Render (free tier available)

1. Create a new **Web Service** on https://render.com
2. Connect your GitHub repo
3. Set:
   - **Root Directory**: *(empty — project root)*
   - **Start Command**: `uvicorn backend.main:app --host 0.0.0.0 --port $PORT`
4. Add environment variables in the Render dashboard

### Option C — AWS EC2 / GCP Compute Engine

```bash
# On the server:
git clone <your-repo>
cd your-project
pip install -r backend/requirements.txt

# Run with gunicorn for production
pip install gunicorn
gunicorn backend.main:app \
  --workers 2 \
  --worker-class uvicorn.workers.UvicornWorker \
  --bind 0.0.0.0:8000
```

Put Nginx in front as a reverse proxy with TLS termination.

## 6. Sequence of first-run API calls

The API has a soft dependency order:

1. `GET /api/health` — confirm dataset is loaded
2. `GET /api/locations` — populate the sidebar dropdown
3. `GET /api/kpis` — populate Overview tab
4. `GET /api/forecast?location=London` — train models, cache result
5. `GET /api/explainability?location=London&model=XGBoost` — requires step 4

Steps 4 and 5 are triggered automatically when the user navigates to those
tabs. The frontend shows a loading spinner with a "first run may take 15–60s"
message while the backend trains models for a new location.
